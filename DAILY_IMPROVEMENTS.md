# PermaCraft — 2026-04-16
## Focus: 🗺️ Map Core — drawing accuracy & snap integrity

Today's target was the map — drawing precision, snap-to-grid fidelity, geodesic measurement, and the panel-to-map navigation loop. After reading `components/map/farm-map.tsx`, `lib/map/snap-to-grid.ts`, `lib/map/zoom-enhancements.ts`, `lib/map/measurement-grid.ts`, `lib/water/calculations.ts`, `components/map/feature-list-panel.tsx`, and the zones API route, I found five self-contained correctness bugs — all shipped.

### 1. Geodesic area for all zone polygons
File: `lib/water/calculations.ts`
What changed: `calculateAreaFromGeometry` used a flat shoelace in raw lng/lat degrees and multiplied by a single `feetPerDegree = 364000`, which only holds at the equator. Because 1° of longitude is `cos(latitude)` narrower than 1° of latitude, the function overstated area by a factor of `1/cos(lat)` — roughly +31% at 40°N, +41% at 45°N, +100% at 60°N. Replaced it with `@turf/area` (already a project dep, used in `boundary-drawer.tsx`) converted from m² to ft² via the exact `SQ_FT_PER_SQ_METER = 10.7639104167`. Added a try/catch and guards for degenerate/empty geometries.
Map/dashboard impact: Zone area display on the Label Zone panel, the water-properties form, and the `/api/farms/[id]/water/calculate-catchment` endpoint all now report real-world square footage. The knock-on effect is that `calculateCatchmentCapture` no longer inflates rainfall capture estimates by the same 30–40% everywhere a designer in the contiguous US places a catchment zone — a permaculture design that actually sizes a cistern or swale off these numbers was silently over-budgeting water.

### 2. Snap-to-grid applies to newly drawn features
File: `components/map/farm-map.tsx`
What changed: Snap was only wired to `draw.update`, so when a user drew a new polygon/line/point at zoom 20+ with snap enabled, the vertices landed wherever the mouse clicked — the "Snap to Grid Enabled" toast was a lie for fresh drawings. Extracted the snap-vertex loop into a local `snapFeatures(features)` helper (reuses the same zoom-gate, touch-aware radius, and farm_boundary guard) and now invoke it from both `handleDrawCreate` and `handleDrawUpdate`. The shared helper also collapsed ~45 lines of duplicated Point/Polygon/LineString snap branches into one.
Map/dashboard impact: A designer working a 50ft urban plot at z20+ can now actually rely on snap for alignment — fence lines meet zone corners, plantings sit on grid centers, swales align to the measurement grid. Previously they had to draw, then immediately drag each vertex to trigger update-time snap.

### 3. Snap grid matches the visual grid at all latitudes
File: `lib/map/snap-to-grid.ts`
What changed: `getGridSpacingDegrees` returned a single scalar that was used for both lat and lng in `getNearestGridIntersection`, but the visual measurement grid correctly uses separate lat/lng degree steps (lng step = `meters / (111320 × cos(lat))`, lat step = `meters / 111320`). At 40°N that's a ~23% lng/lat step mismatch, meaning the magnetic snap target drifted off the visible grid intersection. Introduced a `GridSpacing { latDegrees, lngDegrees }` return type, made `snapCoordinate` / `getNearestGridIntersection` accept either form (legacy scalar still works), added pole-safe cos clamping, and fixed the imperial-to-meters conversion to use the exact `0.3048 m/ft` rather than the inverse approximation. Also unified on the same `111,320 m/°lat` constant the measurement grid uses.
Map/dashboard impact: Snap targets now land on the SAME intersections the user sees drawn on screen. No more "why isn't it snapping to THAT line" at high latitudes. Farms in the contiguous US (25°N–48°N) see snap-drift eliminated in proportion to their latitude.

### 4. Shift-to-disable-snap restores the user's actual preference
File: `components/map/farm-map.tsx`
What changed: On `Shift` release the handler unconditionally called `setSnapToGridEnabled(true)`, so any user who had deliberately toggled snap OFF via the `S` key silently had it re-enabled the next time they held Shift (e.g. typing a capital letter in a nearby field, holding Shift to draw a straight line in a different tool). Switched to a `snapPreShiftRef`/`shiftHeldRef` pair: the preference is captured on the first Shift keydown (ignoring auto-repeat), restored verbatim on keyup, and also restored on window `blur` so users don't get stuck with snap off if they alt-tab while Shift is held. The preference tracker also re-syncs whenever the user toggles via `S` so external changes aren't clobbered.
Map/dashboard impact: `S`-toggle no longer silently reverts. Holding Shift no longer corrupts state if the user releases it outside the window. The "hold Shift to temporarily disable snap" gesture now actually behaves as advertised in `CLAUDE.md`.

### 5. Feature-panel navigation preserves precision context
File: `components/map/feature-list-panel.tsx`
What changed: Clicking any feature in the side panel previously called `flyTo({ zoom: 18 })` — which meant a designer doing precision work at zoom 21 would get yanked OUT of precision mode every time they jumped to a neighboring planting. It also meant clicking a long zone/line centered the map on the centroid and cropped both ends offscreen. Now zones and lines use `fitBounds` on the feature's `@turf/bbox` with 80px padding; both point and area jumps clamp `maxZoom` to `Math.max(18, currentZoom)` so precision mode (z20+) survives panel navigation, and a graceful fallback to `flyTo` handles degenerate geometries. Guarded the `map.getZoom()` call for SSR/test maps that don't implement it.
Map/dashboard impact: A designer comparing two plantings at zoom 21 stays at zoom 21. A long windbreak or swale jumps into view with both ends visible. A tiny point stays readable (no over-zoom past the user's current depth). The feature list now behaves like a proper "navigate to" cockpit instead of a teleporter that breaks your workflow.

## Watch for
- **Stored `estimated_capture_gallons` is now wrong in the other direction.** `app/api/farms/[id]/zones/route.ts` writes a derived `estimated_capture_gallons` at save time using `calculateAreaFromGeometry`. Every row saved before today used the inflated area; after today, new saves use the correct area. We don't auto-recompute. Candidate follow-up: run a one-shot script that loops over `zones` with `catchment_properties IS NOT NULL`, re-parses geometry, and rewrites the JSON. Same story for swale `estimated_volume_gallons` (those use `calculateLengthFromGeometry`, which is already correct, so those rows are fine). Recommend logging a short maintenance script under `scripts/` rather than an inline migration.
- **Grid origin at ± date line / equator crossings.** Both snap and measurement-grid anchor the lattice at (0°, 0°). For any farm whose bounding box straddles 0° (rare for permaculture sites but not impossible in equatorial Africa/Asia/Americas), this still works because `Math.floor(x / step) * step` gives a stable lattice on either side. What *could* break is a farm that crosses the antimeridian (±180°); no existing check — but this has been the case since the grid shipped, so not a regression.
- **Zone POST does DELETE-then-batch-INSERT across two executions.** `app/api/farms/[id]/zones/route.ts:79-148` runs the `DELETE FROM zones WHERE farm_id = ?` as its own `db.execute`, then issues `db.batch(statements)` separately. If the batch insert ever fails (malformed geometry, batch size cap, transient Turso error), the farm ends up with ZERO zones persisted. Not a map-core fix — schema-safe but needs a transaction wrapper (either `db.batch` both operations together as one atomic batch, or use a try/catch that restores from an in-memory snapshot). Flagging for a future ⚡ Performance/Reliability day.
- **snapFeatures uses `farm.center_lat`; measurement-grid uses farm-bounds centroid.** For typical farms these are within ~0.01°, which is noise relative to the grid step. Only matters for large (>5-mile) farms — worth a unit test if we ever add one.
- **`@turf/area` adds ~5 KB to the bundle that wasn't there in `calculations.ts`** (it was already imported by `boundary-drawer.tsx`, so it was already in the chunk graph). Accuracy is worth it — flagging only if bundle budgets get tight.
